# Product Management

Spring Boot 기반의 상품 관리 프로젝트입니다.  
상품 등록, 조회, 수정, 삭제 기능을 구현하면서 계층형 구조, 의존성 주입, 리팩토링, 테스트 전략을 함께 학습하는 것을 목표로 진행했습니다.
jpa같은 ORM을 쓰지않고 직접 jdbctemplate을 이용해 데이터베이스에 직접 접근하는 방식으로 구현하여 SQL과 데이터 매핑 과정을 이해하는데 집중했습니다.

## 프로젝트 목표
- 상품 관리 기능을 Spring 기반으로 구현하기
- 계층 분리를 통해 유지보수하기 쉬운 구조 만들기
- 테스트 코드를 기반으로 안정적으로 리팩토링하기
- 실행 환경에 따라 다른 의존성을 주입하는 방법 익히기

## 주요 학습 내용

### 1. 추상화와 의존성 주입
`ProductRepository` 인터페이스를 중심으로 여러 구현 클래스를 추상화하면서, 서비스 계층이 구체 클래스가 아닌 추상화에 의존하도록 설계하는 방법을 학습했습니다.  
이 과정을 통해 의존성 주입 패턴과 의존성 역전 원칙(DIP)의 의미를 이해했습니다.

### 2. 리팩토링과 객체 변환 로직 개선
기존에는 `ModelMapper`를 사용해 DTO와 Entity를 변환했지만, 이후 `toEntity()`, `toDto()` 메서드를 직접 구현하는 방식으로 리팩토링했습니다.  
이를 통해 객체 변환 과정을 더 명확하게 제어하고, 코드의 의도를 분명히 표현하는 방법을 익혔습니다.

### 3. Profile을 활용한 환경 분리
`Profile`을 사용하여 실행 환경에 따라 서로 다른 의존성을 생성하고 주입하는 방법을 학습했습니다.  
개발, 테스트, 운영 환경에 맞는 설정을 분리하면서 환경별 실행 전략의 필요성을 이해했습니다.

### 4. 테스트 기반 리팩토링
테스트 코드를 먼저 작성하고 이를 기준으로 기능을 검증하면서 리팩토링을 진행했습니다.  
이 과정을 통해 코드 변경 이후에도 기존 기능이 정상 동작하는지 빠르게 확인할 수 있었고, 더 안정적으로 구조를 개선할 수 있었습니다.

### 5. 단위 테스트와 통합 테스트
단위 테스트와 통합 테스트를 각각 작성하면서 두 테스트 방식의 목적과 차이를 학습했습니다.
- 단위 테스트는 Mock 객체를 활용해 서비스 로직을 빠르게 검증하는 데 사용했습니다.
- 통합 테스트는 스프링 컨텍스트와 데이터베이스를 포함한 실제 실행 흐름을 검증하는 데 사용했습니다.

## 테스트
이 프로젝트에서는 다음 두 가지 방식의 테스트를 작성했습니다.
- 단위 테스트: 외부 의존성을 Mock으로 대체하여 핵심 비즈니스 로직만 검증
- 통합 테스트: 스프링 빈 주입, DB 연동 등 실제 실행 환경에 가까운 흐름 검증

테스트를 통해 기능 구현뿐 아니라 리팩토링 이후의 안정성도 함께 확인했습니다.

## 기술 스택

- Java
- Spring Boot
- Spring JDBC
- JUnit 5
- Mockito
- MySQL

## 느낀 점
이번 프로젝트를 통해 기능 구현 자체보다, 구조를 어떻게 설계하고 어떻게 검증할 것인지가 더 중요하다는 점을 배웠습니다.  
특히 추상화, 의존성 주입, 리팩토링, 테스트 코드 작성은 서로 분리된 개념이 아니라 함께 사용될 때 더 큰 효과를 낸다는 점을 경험할 수 있었습니다.
또한 학습 목적상 JPA를 사용하지 않고 JdbcTemplate과 NamedParameterJdbcTemplate을 활용해 데이터베이스에 직접 접근하는 방식으로 구현했습니다. 
이를 통해 ORM에 의존하기 전에 SQL 실행, 파라미터 바인딩, 조회 결과 매핑, 레포지토리 계층의 역할을 직접 이해할 수 있었습니다.
지금까지는 LLM을 이용해 mvp를 급급하게 만든느데만 치중했지만, 실제 프레임워크의 동작 방식, 협업을 위한 최소한의 상식(단위 테스트, 통합 테스트 등)을 배우는데 집중했습니다.
